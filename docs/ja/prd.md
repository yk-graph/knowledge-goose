# プロダクト要件定義: 競合カフェ分析・可視化ツール

> Available in: [English](../prd.md) | 日本語
>
> 本ドキュメントは、本プロダクトの要件を定義する。設計判断の背景・代替案・採用理由は別途開発ブログに記録する(本書の scope 外)。

---

## 1. プロダクト概要

**バンクーバーの 1 店舗のカフェ経営者が、近隣競合との比較からデータに基づく打ち手を導けるようにする、競合情報の記録・可視化ツール。**

### 誰の / 何を / どう解決するか

- **誰の**: バンクーバーで営業する 1 店舗のカフェ経営者
- **何を**: 来店を増やしたいが、判断材料がなく何をすべきか分からないという課題を
- **どう解決**: 近隣競合店の情報(店舗特性・観察記録・Google 公開情報)を一元管理し、自店と視覚的に比較できるツールを提供する

### プロダクトの位置付け

- シングルテナント設計(対象は固定の 1 店舗 = 自店舗)
- 調査員が手動でデータを記録する前提
- 本アプリの責務は「入力プラットフォーム」と「可視化ツール」のみ

### Scope 外

- 調査員の派遣・運用設計
- 収集データの精度担保
- 多店舗展開・マルチテナント対応

### 主機能

- **MAP**: 競合店をピン表示、クリックで詳細
- **時系列グラフ**: レビュー数・客数・想定売上の推移
- **比較テーブル**: 項目別の自店 vs 競合

## 2. ターゲットユーザー

### プライマリユーザー: カフェオーナー / 経営者

- **役割**: バンクーバーで営業するカフェ 1 店舗のオーナーまたは経営層
- **特性**:
  - 業務時間の大半を現場運営に充てており、分析に割ける時間は限られる
  - 高度なデータ分析の専門性は持たない前提
  - 経営判断の最終決裁者
- **使用デバイス**: PC(店舗のバックオフィス、自宅)
- **使用シーン**: 経営判断のための定期分析(週 1〜月 1)、新施策の検討時
- **求めるもの**: 競合との差分が一目で分かる可視化、データに基づく仮説立案の補助
- **許容できないもの**: 判断のための情報を集めるのに時間がかかる体験

### セカンダリユーザー: データ入力を担う調査員

- **役割**: オーナーから委託され、競合店を訪問して観察データを記録する人
- **使用デバイス**: スマートフォン(現地)/ PC(後日まとめて入力)
- **使用シーン**: 競合店訪問中、または訪問後のデータ入力
- **本 MVP での扱い**: UI は最低限。多少の不便は許容される前提(オーナーの依頼で使う立場のため)

### ユーザー数の前提

- プライマリ: **1 名**(オーナー)
- セカンダリ: 数名規模(同店の調査員)
- 全員が個別アカウントを持ち、認証して利用する
- データ自体は単一店舗のもの(シングルテナント設計は維持)

### 認証・認可

- **認証必須**: 未認証ユーザーはアプリにアクセスできない
- **認証方法**: 以下の 2 種類のみ
  - メールアドレス + パスワード
  - Google アカウント(OAuth)
- **対応しないもの**: Twitter / X、Facebook、GitHub、Apple など他の SNS OAuth プロバイダ
- **認可モデル**: ロールベース、2 ロール(`admin` / `staff`)

### ロール定義

- **`admin`**(想定: カフェオーナー / 経営者)
  - 自店マスタの編集
  - 競合マスタの追加・編集・削除
  - 観察記録の入力・編集・削除(全レコード)
  - 全可視化の閲覧
  - ユーザー一覧の閲覧
- **`staff`**(想定: 調査員)
  - 観察記録の入力(自分が入力したものは編集可、他人のは閲覧のみ)
  - 全可視化の閲覧
  - 自店・競合マスタは閲覧のみ
  - ユーザー一覧は閲覧不可

### 初期 admin の設定

- 新規登録は全員 `staff` ロール
- `admin` 付与は実装者が DB を直接 SQL で更新する(ロール変更 UI は MVP に含めない)

### 設計上の含意

- 可視化の習熟コストはゼロに近い必要がある(慣れない経営者も理解できる)
- 専門用語は避け、見たままで意味が分かる UI を志向
- 入力 UI は機能優先、デザインの磨き込みは後回し
- 認証実装のため MVP スコープが拡大する(0.5〜1 日分の作業を確保)

## 3. ユーザーストーリー

形式: `[ID] [優先度] 〜として、〜したい。なぜなら〜`

優先度: **MVP**(初回リリースに含める)/ **後回し**(MVP 後に着手)

### Epic A: 認証

- **`US-A1` MVP** ── ユーザーとして、メールアドレス + パスワード、または Google アカウントで **新規登録** したい。なぜなら、本アプリは認証必須でアクセスを限定する必要があるから。
- **`US-A2` MVP** ── ユーザーとして、登録済みアカウントで **ログイン・ログアウト** したい。なぜなら、自分のセッションを管理し、他のユーザーと区別する必要があるから。
- **`US-A3` MVP** ── ユーザーとして、パスワードを忘れた場合に **メール経由で再設定** したい。なぜなら、再登録なしでアカウントを継続利用できる必要があるから。

### Epic B: 自店マスタ管理

- **`US-B1` MVP** ── `admin` として、自店舗の情報(基本情報、Google Place ID 紐付け、平均単価などの手入力項目)を **登録・編集** したい。なぜなら、競合との比較の基準となる自店データが必要だから。

### Epic C: 競合店マスタ管理

- **`US-C1` MVP** ── `admin` として、Google Places API で店名検索し、ヒットした競合店を **ワンクリックで追加** したい。なぜなら、住所・営業時間・写真などを手入力する手間を省きたいから。
- **`US-C2` MVP** ── `admin` として、競合店の **手入力項目(平均単価、席数、席構成、SNS など)を編集** したい。なぜなら、Google から取れない情報を補完する必要があるから。
- **`US-C3` 後回し** ── `admin` として、追跡をやめた競合店を **削除** したい。なぜなら、不要なノイズを比較から除外したいから。

### Epic D: 観察記録の入力

- **`US-D1` MVP** ── `staff` として、競合店訪問時の **観察データ(客数、客層、利用シーン、天候など)を入力** したい。なぜなら、時系列で集客の変化を追えるようにする必要があるから。
- **`US-D2` 後回し** ── `staff` として、自分が入力した観察データを **編集** したい。なぜなら、入力ミスや追加情報を反映する必要があるから。
- **`US-D3` 後回し** ── `admin` として、**すべての観察データを編集・削除** したい。なぜなら、データ品質の最終責任を持つから。

### Epic E: MAP 可視化

- **`US-E1` MVP** ── ユーザーとして、自店と競合店を **マップ上にピン表示** で確認したい。なぜなら、地理的な競合状況を一目で把握したいから。
- **`US-E2` MVP** ── ユーザーとして、ピンをクリックして **店舗の詳細情報を確認** したい。なぜなら、特定店舗の情報に素早くアクセスしたいから。

### Epic F: 時系列グラフ

- **`US-F1` MVP** ── ユーザーとして、自店と競合店の **客数・想定売上の推移** を比較したい。なぜなら、集客パターンの違いを読み取りたいから。
- **`US-F2` 後回し** ── ユーザーとして、**Google レビュー数・評価の推移** を時系列で確認したい。なぜなら、評価の変化と集客の関係を把握したいから。

### Epic G: 比較テーブル

- **`US-G1` MVP** ── ユーザーとして、自店と選択した競合店の **項目別比較を表で** 確認したい。なぜなら、価格・席数・営業時間などの差異を一覧で把握したいから。

### 集計

| カテゴリ | 件数      |
| -------- | --------- |
| MVP      | 10 件     |
| 後回し   | 4 件      |
| **合計** | **14 件** |

## 4. 受け入れ基準

「後回し」ストーリーの基準は MVP 範囲外として割愛し、**MVP 10 件のみ** 定義する。

### Epic A: 認証

#### `US-A1` 新規登録

- [ ] メールアドレス + パスワードで新規登録できる
- [ ] パスワードは **8 文字以上** で英字と数字をそれぞれ含む
- [ ] Google アカウントでサインアップできる
- [ ] 既に登録済みのメールアドレスでの再登録はエラーになる
- [ ] 新規登録(メール+パスワード)時、Resend 経由で確認メールが送信される
- [ ] 確認メール内のアクティベーションリンクをクリックして初めてアカウントが有効になる
- [ ] アクティベーション完了前のアカウントではログインできない(エラー表示)
- [ ] アクティベーションリンクは 24 時間で失効する
- [ ] 「確認メールを再送信」機能を持つ
- [ ] Google アカウントでサインアップした場合は、メール確認フローをスキップする
- [ ] 新規登録完了時、ロールは **`staff`** が自動付与される

#### `US-A2` ログイン / ログアウト

- [ ] 登録済みメール + パスワードでログインできる
- [ ] Google アカウントでログインできる
- [ ] 不正な認証情報の場合、入力欄付近にエラーメッセージが表示される
- [ ] ログアウト操作でセッションが破棄され、ログイン画面に遷移する
- [ ] 未ログイン状態でアプリの保護画面にアクセスすると、ログイン画面にリダイレクトされる

#### `US-A3` パスワードリセット

- [ ] ログイン画面に「パスワードを忘れた方」リンクがある
- [ ] メールアドレスを入力すると、Resend 経由でリセットメールが送信される
- [ ] 未登録メールでも同じ「メールを送信しました」表示(列挙攻撃の防止)
- [ ] リセットメール内のリンクは 1 時間で失効する
- [ ] リンクから遷移した画面で新パスワードを設定できる(8 文字以上、英字 + 数字)
- [ ] 設定完了後はログイン画面に遷移する(自動ログインはしない)
- [ ] Google アカウント認証ユーザーには「Google アカウントの設定からパスワードを変更してください」と案内する

### Epic B: 自店マスタ管理

#### `US-B1` 自店マスタ登録・編集

- [ ] `admin` ロールのみ自店登録・編集画面にアクセスできる(`staff` は閲覧のみ可)
- [ ] **自店(`is_own_store = true`)は 1 件のみ登録可能**。2 件目を作ろうとするとエラー
- [ ] Google Place ID を入力 / 検索すると、Google Places API から取得した情報がプレビュー表示される
- [ ] 手入力項目(`average_price`, `seat_count`, `menu_count`, `seat_composition`)を入力できる
- [ ] 必須項目(`name`, `address`, `average_price`)が未入力の場合、保存できない
- [ ] 保存成功後、MAP / 比較テーブルなど可視化に「自店」として反映される

### Epic C: 競合店マスタ管理

#### `US-C1` Google Places API で競合店をワンクリック追加

- [ ] `admin` ロールのみアクセスできる
- [ ] 店名(自由テキスト)で検索でき、結果が一覧表示される
- [ ] 検索結果は **バンクーバー周辺にバイアス** がかかっている(`locationBias` 指定)
- [ ] 検索結果からワンクリックで競合店を追加できる
- [ ] 追加時に Google データ(name, address, location, hours, rating, review count, photos, price level, categories)が自動入力される
- [ ] 既に追加済みの店舗(同じ `google_place_id`)は重複登録できない(エラー表示)

#### `US-C2` 競合店の手入力項目編集

- [ ] `admin` ロールのみ編集画面にアクセスできる
- [ ] 手入力項目(`average_price`, `seat_count`, `menu_count`, `seat_composition`, SNS handle)を編集できる
- [ ] 「Google データ再同期」ボタンで最新の Google 情報を取り込み直せる
- [ ] 必須項目(`average_price`)が未入力の場合、保存できない
- [ ] 保存成功後、可視化に反映される

### Epic D: 観察記録の入力

#### `US-D1` 観察記録の入力

- [ ] 認証済みユーザー(`admin` / `staff` 両方)がアクセスできる
- [ ] 対象店舗(自店 or 競合)を選択して観察を記録できる
- [ ] 必須項目: `observed_at`, `observation_duration_minutes`, `customer_count`, `gender_ratio_male`, `gender_ratio_female`, `age_distribution`, `observed_usage_scene`, `weather`
- [ ] `gender_ratio_male + gender_ratio_female = 100` でない場合、保存エラー
- [ ] `age_distribution` の合計が 100 でない場合、保存エラー
- [ ] 保存時に **`observer_id` が現在のログインユーザーに自動紐付け** される
- [ ] 保存時に **`day_of_week` / `time_band` が `observed_at` から自動算出** される
- [ ] 保存時に **Google スナップショット(`google_rating_snapshot`, `google_review_count_snapshot`)が現時点の値で記録** される

### Epic E: MAP 可視化

#### `US-E1` 競合店マップ表示

- [ ] 認証済みユーザーがアクセスできる
- [ ] 自店は **専用のピン色 / アイコン** で強調表示される
- [ ] 競合店は別の色 / アイコンで表示される
- [ ] **全ピンが画面内に収まるよう自動 zoom-fit** される(初期表示)
- [ ] 店舗が 1 件も登録されていない場合は、その旨と追加導線が表示される

#### `US-E2` ピンクリックで店舗詳細表示

- [ ] ピンクリックでポップアップまたはサイドパネルが開く
- [ ] 詳細情報に最低限以下を含む: `name`, `address`, `business_hours`, `google_rating`, `google_review_count`, `average_price`, `seat_count`, 写真 1 枚以上
- [ ] 詳細から「比較テーブルに追加」アクションを起動できる(`US-G1` への導線)

### Epic F: 時系列グラフ

#### `US-F1` 客数・想定売上の時系列グラフ

- [ ] 認証済みユーザーがアクセスできる
- [ ] 自店 1 件 + 競合店 1 〜 3 件、合計最大 4 件を選択して比較表示できる
- [ ] 期間切替 UI を持つ(直近 1 週間 / 1 ヶ月 / 3 ヶ月 / 任意期間)
- [ ] 指標切替 UI を持つ(客数 / 想定売上 を切り替え可能)
- [ ] データがない期間は空白として表示する(0 と誤認させない)
- [ ] 凡例が表示され、自店と競合が視覚的に区別できる

### Epic G: 比較テーブル

#### `US-G1` 項目別の比較テーブル

- [ ] 認証済みユーザーがアクセスできる
- [ ] 自店 1 件 + 競合店 1 〜 3 件、合計最大 4 件を選択して比較表示できる
- [ ] 比較項目に最低限以下を含む: `average_price`, `seat_count`, `menu_count`, `seat_composition`, `business_hours`, `google_rating`, `google_review_count`, `google_categories`
- [ ] 自店と競合の差が一目で分かる視覚表現(色 / 矢印 / ハイライト)を持つ
- [ ] 列の追加・削除など項目選択 UI を持つ(MVP では固定セットでも可)

## 5. MVP に含めない項目(Scope 外)

これまでの議論で確定した scope 外項目を集約する。**3 つのカテゴリ** に分類:

- **A. 機能として後回し** ── 機能としては有意義で、MVP リリース後に追加する可能性がある
- **B. プロダクト方向性として scope 外** ── 本プロダクトの設計思想として最初から含めない
- **C. 非機能要件として後回し** ── パフォーマンス・運用面の高度化は MVP 後に検討

### A. 機能として後回し

| 項目 | 関連 ID / 出典 |
| --- | --- |
| 競合店の削除機能 | `US-C3` |
| 観察記録の編集機能(staff: 自分のレコード) | `US-D2` |
| 観察記録の編集・削除機能(admin: 全レコード) | `US-D3` |
| Google レビュー数・評価の推移グラフ | `US-F2` |
| 比較テーブルの CSV エクスポート | 受け入れ基準議論時 |
| ユーザー管理 UI(招待・ロール変更・削除) | 認証議論時(`admin` ロール付与は SQL 直接更新で対応) |
| 比較項目のユーザーカスタマイズ(MVP では固定セット) | `US-G1` |
| レーダーチャート可視化 | 可視化機能選定時 |
| 過去データの CSV 一括 import | — |

### B. プロダクト方向性として scope 外

本プロダクトの設計思想に含まれない範囲。**永続的に scope 外** と扱う。

- **データ収集オペレーション**
  - 調査員の派遣・採用・教育
  - 観察データの精度担保プロセス
  - 自動データ収集(Popular Times のスクレイピング含む。Google 公式 API で取得できないため)
- **プラットフォーム拡張**
  - マルチテナント(複数経営者が同一インスタンスを共有)
  - 多店舗展開(自店は 1 店舗固定)
  - 多通貨対応(CAD 固定)
  - モバイルアプリ(iOS / Android ネイティブ)
- **認証プロバイダ**
  - Twitter / X、Facebook、GitHub、Apple、その他の SNS による OAuth(メール + Google のみ)

### C. 非機能要件として後回し

- リアルタイム同期(WebSocket / Server-Sent Events)
- オフライン対応 / PWA 化
- Push 通知
- 高度なパフォーマンス最適化(CDN 戦略、エッジキャッシュなど)
- SEO 対策(認証必須のため不要だが、念のため明記)
- 監査ログの UI 可視化(DB に持つが、UI では見せない)
- データのバックアップ・復元機能(運用フェーズでの検討事項)

### 次期リリース候補(Post-MVP 高優先)

3 日 MVP のリリース後、**最優先で追加検討する機能**。実利用上の必要性が高く、portfolio 観点でも「実利用の声を反映した進化」を語れる。

- **多言語対応(i18n: 日本語 + 韓国語)**
  - **背景**: 対象店舗のオーナーが韓国人であり、調査員に日本語ネイティブが含まれる可能性があるため
  - **実装規模**: i18n フレームワーク導入 + 全 UI 文言の外部化、約 1 日
- **Excel / スプレッドシート形式でのエクスポート**
  - **背景**: 経営会議では Excel / Google Sheets で資料整理する文化があり、コピー&ペーストでは効率が悪いため
  - **実装規模**: 比較テーブル + 観察データの xlsx 出力、約 0.5 日
  - MVP の単純 CSV エクスポート(後回し A)とは別物として位置付ける(整形済みフォーマット)

## 6. リリース基準 / Definition of Done

3 日 MVP として「リリース完了」と宣言するための **必達条件**。これを満たさない限り MVP は完成と見なさない。

### 機能要件
