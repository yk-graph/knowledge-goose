# 作業手順アウトライン: ClaudeCode で 3 日でアプリをリリースする

> Available in: [English](../workflow-checklist.md) | 日本語
>
> **方針**: 自分ではソースコードは書かず、CC に書かせる。コードの意図と意味は理解し、間違いがあれば訂正を指示する。

---

## 全体スケジュール(目安)

| Day                 | フェーズ               | 時間配分     |
| ------------------- | ---------------------- | ------------ |
| Day 1 午前          | Phase 1: 要件定義      | 2〜3 時間    |
| Day 1 午後          | Phase 2: 設計          | 3〜4 時間    |
| Day 1 夜            | Phase 3: CC 足場づくり | 1〜2 時間    |
| Day 2 午前          | Phase 4: タスク分解    | 1〜2 時間    |
| Day 2 〜 Day 3 午前 | Phase 5: 実装ループ    | メインの時間 |
| Day 3 午後          | Phase 6: デプロイ      | 2〜3 時間    |
| Day 3 夜            | バッファ・リリース告知 | —            |

---

## Phase 1: 要件定義(CC を起動する前)

ツール: Claude.ai か紙。CC はまだ起動しない。

- [ ] アプリのアイデアを 3 行で書き出す(何を / 誰のために / なぜ)
- [ ] `docs/prd.md` を作成
  - [ ] プロダクト概要
  - [ ] ターゲットユーザー
  - [ ] ユーザーストーリー(「〜として、〜したい、なぜなら〜」形式で 3〜7 個)
  - [ ] 受け入れ基準(チェックリスト形式)
  - [ ] **MVP に含めない項目(スコープ外)** ← 重要
  - [ ] 成功指標(リリース後に何で評価するか)

**完了の判定**: PRD を読めば、自分以外でも何を作るか理解できる状態。

---

## Phase 2: 設計

ツール: Claude.ai。Phase 2 の最後で CC の Plan mode を試す。

- [ ] `docs/stack.md` を作成
  - [ ] フレームワーク / UI / 状態管理 / テスト / デプロイ先を箇条書き
  - [ ] 各選定の理由を一行
- [ ] `docs/architecture.md` を作成
  - [ ] 画面一覧
  - [ ] 画面遷移図(mermaid)
  - [ ] コンポーネント階層
- [ ] `docs/data-model.md` を作成
  - [ ] TypeScript の `interface` で型定義
  - [ ] バックエンドがあれば API endpoint 一覧
- [ ] `docs/conventions.md` を作成
  - [ ] 命名規則
  - [ ] ディレクトリ構成
  - [ ] import 順
  - [ ] テスト方針
  - [ ] コミット規約 / ブランチ戦略
- [ ] (任意)CC を起動して Plan mode で設計レビューさせる

**完了の判定**: 上記 4 ファイルがあり、設計上の疑問が解消している。

## Phase 3: CC 足場づくり ★ ここが本丸

ツール: CC をいよいよ起動する。

### リポジトリ初期化

- [ ] git リポジトリ作成、`main` ブランチで初期コミット
- [ ] フレームワークのプロジェクト雛形を生成
- [ ] `.gitignore`、`.editorconfig`、`.nvmrc` などの基本ファイル
- [ ] `README.md` の骨組み

### CC が読む常時ファイル

- [ ] `CLAUDE.md` をリポジトリ直下に作成
  - [ ] プロジェクト概要(2〜3 行)
  - [ ] ディレクトリ構成のサマリ
  - [ ] ビルド / テスト / Lint / 型チェックのコマンド
  - [ ] コーディング規約の要点(詳細は `docs/conventions.md` を参照、と書く)
  - [ ] よくある罠 / 注意点
  - [ ] **大きくなりすぎないこと**(目安 200 行以内)

### カスタムスラッシュコマンド

- [ ] `.claude/commands/` ディレクトリを作成
- [ ] `feature.md`(新機能テンプレ)
- [ ] `review.md`(自己レビュー)
- [ ] `fix-tests.md`(失敗テストの修復)
- [ ] `refactor.md`(リファクタ指針)
- [ ] その他、自分のワークフローでよく使う指示があれば追加

### MCP / Skills

- [ ] 必要なら `.mcp.json` を設定(GitHub / Figma など)
- [ ] 必要なら `.claude/skills/` に専用作法ファイルを配置

### 開発環境の整備

- [ ] Lint(ESLint など)が `npm run lint` で動く
- [ ] Formatter(Prettier など)が `npm run format` で動く
- [ ] テスト(Vitest / Jest など)が `npm run test` で動く
- [ ] 型チェック(`tsc --noEmit`)が `npm run typecheck` で動く
- [ ] Git hooks(husky + lint-staged)で commit 前にチェック
- [ ] **ここまでで一度コミット**

**完了の判定**: 新しく `claude` を起動してタスクを渡しても、CC が規約とコマンドを把握した状態で動ける。

## Phase 4: タスク分解

ツール: Claude.ai か CC、どちらでも。

- [ ] PRD のユーザーストーリーをエピックに分解
- [ ] エピックをタスクに分解
  - [ ] 各タスクは **1 PR で終わる粒度**
  - [ ] 各タスクに「完了の定義」を書く
  - [ ] 依存関係を矢印でメモ
- [ ] `TODO.md` か GitHub Issues に登録
- [ ] マイルストーン(MVP / α / β)を設定
- [ ] 最初のマイルストーンに含めるタスクを確定

**完了の判定**: 最初に着手するタスクが明確で、迷わず Phase 5 に入れる。

---

## Phase 5: 実装ループ(タスクごとに繰り返す)

各タスクで以下を 1 セッションとして実行:

- [ ] `/clear` でコンテキストをリセット
- [ ] タスクの仕様を貼る(PRD と設計書の該当部分)
- [ ] **Plan mode (Shift+Tab)** で計画させる
- [ ] 計画をレビュー、必要なら修正指示
- [ ] 通常モードに戻して実装
- [ ] テストを追加(または書かせる)
- [ ] テスト実行 → 失敗したらログを CC に戻して自己修正
- [ ] 型チェック / Lint をパス
- [ ] `/review` で自己レビュー
- [ ] 自分でも diff を読む(コードの意図と意味を理解、必要なら訂正指示)
- [ ] コミット → PR
- [ ] CI が緑になることを確認
- [ ] マージ
- [ ] `TODO.md` のチェックを更新

### コンテキスト管理(セッション中)

- [ ] 長くなったら `/compact` で要約
- [ ] それでも辛ければ `/clear` でリセット(必要情報は CLAUDE.md か作業メモに退避)
- [ ] 並列で進めたいときは `git worktree add` + 別 CC セッション

### 詰まったときのチェックリスト

- [ ] タスク粒度は適切か(大きすぎないか)
- [ ] CC に渡した仕様は具体的か
- [ ] テストが書けない設計になっていないか
- [ ] Plan mode を飛ばしていないか
- [ ] CLAUDE.md の内容と現実が乖離していないか

  - [ ] 起動方法
  - [ ] デプロイ方法
